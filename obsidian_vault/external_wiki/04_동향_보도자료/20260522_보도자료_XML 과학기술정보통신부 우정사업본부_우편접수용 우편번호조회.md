# 핵심 요약
- 금융회사 및 IT 기업이 우정사업본부의 우편번호조회 오픈 API를 시스템에 연계할 때 발생할 수 있는 보안 취약점과 컴플라이언스 위험을 선제적으로 예방하기 위한 통제 방안임.
- REST/XML 방식의 외부 API 통신 과정에서 발생 가능한 망분리 우회 위협을 차단하고, 연계 구간의 안전성을 확보하는 데 목적이 있음.
- 고객의 주소 정보 수집·처리 시 개인정보 보호법상 안전성 확보조치 기준 및 금융보안원 가이드를 준수하도록 가이드라인을 제공함.

# 주요 통제 및 규제 사항
- **외부 연계 통제 및 망분리 준수 (전자금융감독규정 제15조)**: 금융회사의 내부망에서 외부 공공 API망(우정사업본부)으로 직접 아웃바운드 연결을 수행할 수 없음. 반드시 DMZ 구간에 위치한 API 게이트웨이 또는 연계 프록시(Proxy) 서버를 경유하도록 네트워크 아키텍처를 설계하고 통제해야 함.
- **연계 데이터 암호화 및 개인정보 보호 (개인정보 보호법 제29조)**: 조회 대상인 우편번호 자체는 개인식별성이 낮으나, 사용자가 입력하는 상세 주소 및 고객 식별 정보와 결합 시 개인정보로 분류됨. 연계 전송 구간은 반드시 HTTPS(TLS 1.2 이상) 프로토콜을 적용하여 데이터 위변조 및 스니핑을 방지해야 함.
- **인증정보 관리 및 하드코딩 금지 (ISMS-P 인증기준 2.6.2)**: 오픈 API 호출에 사용되는 서비스 키(인증키)는 소스코드 내에 하드코딩해서는 안 되며, 보안성이 확보된 환경 변수 설정 파일이나 KMS(Key Management System)를 통해 주입 및 관리해야 함.
- **트래픽 제어 및 서비스 가용성 보장**: 개발계정(일 10,000건 제한) 및 운영계정의 트래픽을 실시간 모니터링하고, 외부 API 장애 발생 시 내부 금융 서비스 전체가 마비되지 않도록 서킷 브레이커(Circuit Breaker) 패턴을 적용하여 가용성을 보장해야 함.

# 실무 대응 방향
- **API 연계 아키텍처 재구축**: 내부망 WAS가 직접 외부 API를 호출하는 경로가 있는지 전수 점검하고, '내부망 - API 게이트웨이(DMZ) - 우정사업본부 외부망' 형태로 통신 경로를 단일화하고 방화벽 룰셋을 최소화하여 운영함.
- **시크릿 관리 프로세스 표준화**: 소스코드 형상관리 시스템(Git 등)에 외부 API 인증키가 노출되지 않도록 사전 차단 도구(Git-secrets 등)를 파이프라인에 이식하고, 주기적으로 API 인증 키를 로테이션하는 내부 프로세스를 수립함.
- **장애 대응 및 Fallback 메커니즘 설계**: 우정사업본부 API 서버의 일시적 장애나 트래픽 초과로 인한 응답 지연 발생 시, 사용자 화면에서 수동 주소 입력으로 즉시 전환될 수 있는 예외 처리(Fallback) 로직을 프론트엔드와 백엔드에 각각 구현함.
- **연계 로그 보존 및 모니터링**: 외부 API 호출 시 발생하는 요청/응답 로그를 최소 1년 이상 보존하되, 로그 내에 상세 주소 등 민감한 개인정보가 평문으로 기록되지 않도록 마스킹 필터링 조치를 적용함.
